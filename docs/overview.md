# Project Overview

## 목적
이미지 파일과 설명 JSON이 한 쌍으로 저장되는 폴더를 감시하고, 자동으로 MongoDB와 MinIO에 적재하는 시스템을 만든다.

이 프로젝트의 목적은 다음 네 가지다.
- 메타데이터 기반 빠른 조회
- 이미지 원본의 안정적 저장과 다운로드
- 필터링 가능한 탐색 UI
- 대량 데이터에서도 유지 가능한 구조 확보

## 워크스페이스 구조
- 새 백엔드 구현은 `apps/backend/`에서 시작한다.
- 최종 프론트엔드 구현은 `apps/frontend/`의 SvelteKit에서 진행한다.
- Electron 데스크톱 앱은 `apps/desktop/`에 둔다.
- 공용 타입과 계약은 `packages/shared/`로 모은다.
- Docker 배포 파일은 `infra/docker/`에서 관리한다.
- 기존 프로토타입은 `legacy/minio-prototype/`에 보관한다.

## 아키텍처
투트랙 구조를 사용한다.

- MongoDB
  - 목록 조회
  - 필터 검색
  - 메타데이터 조회
  - 페이징
- MinIO
  - 이미지 원본 저장
  - 이미지 다운로드
  - 이미지 표시용 객체 제공

## 데이터 흐름
1. 특정 폴더에 `image + json` 쌍이 저장된다.
2. 백엔드가 폴더 변화를 감지하거나 수동 스캔한다.
3. JSON에서 메타데이터와 태그를 추출한다.
4. MongoDB에 검색용 문서를 저장한다.
5. MinIO에 이미지 원본을 저장한다.
6. 프론트는 backend API를 통해 MongoDB 기준 목록을 조회하고 MinIO 기준 이미지를 본다.

## 핵심 원칙
- 메타데이터와 태그는 분리 저장하지 말고 하나의 metadata 문서로 합친다.
- 검색의 정본은 MongoDB로 둔다.
- 바이너리 파일의 정본은 MinIO로 둔다.
- UI는 이미지 뷰어이자 탐색기여야 한다.

## 주요 기능
- 이미지 목록 조회
- 페이지네이션
- 필터링
  - 태그
  - 공정코드
  - 판정 결과
  - 날짜 범위
  - 기타 검색 필드
- 이미지 단건 보기
- 체크리스트 기반 다중 다운로드
- 메타데이터 확인

## 비기능 요구
- 25,000건 수준에서 목록/필터가 유지되어야 한다.
- 파일 조회와 메타데이터 조회는 분리해서 측정할 수 있어야 한다.
- 화면은 대량 데이터에서도 멈춘 것처럼 보이지 않아야 한다.
- 로딩 상태와 빈 상태를 명확히 보여야 한다.
